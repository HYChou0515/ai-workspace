# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Language

Respond to the user in **Traditional Taiwanese Chinese (繁體中文 / 台灣用語)**. Keep code, identifiers, commit messages, and file contents in English unless the user explicitly asks otherwise.

## Workflow

- **New feature requests and bug reports**: start with **`/grill-me`** to stress-test the plan and resolve open questions before any code is written.
- **Implementation**: once the plan is clear, use **`/tdd`** to drive the work through the red-green-refactor loop rather than writing implementation first.
- **Phase numbering**: plans use a **flat integer sequence** — `Phase 1`, `Phase 2`, … (`P1`, `P2`, …). **Never** sub-phases like `Phase 1a` / `Phase 1b`. "Phase 1" means Phase 1 is to be *completed*; if work is split off, it becomes the next integer (`Phase 2`), not a letter suffix.

## Architecture

Pluggable layers connected through Protocols — swap any single piece by writing a new implementation and injecting it into `create_app`.

```
React SPA (web/) ─► FastAPI app (api/) ─► OpenAI Agents SDK (api/litellm_runner.py)
                          │                       │
                          │                       └─► LiteLLM ─► Ollama / hosted LLM
                          │
                          ├─► AgentRunner Protocol (api/runner.py)
                          │     - LitellmAgentRunner: real LLM, see above
                          │     - ScriptedAgentRunner: scripted events, used by tests
                          │
                          ├─► Sandbox Protocol (sandbox/protocol.py)
                          │     - MockSandbox: in-memory, for tests
                          │     - LocalProcessSandbox: subprocess + temp dir, default for VM deploys
                          │
                          ├─► FileStore Protocol (filestore/protocol.py)
                          │     - SpecstarFileStore: per-workspace blob inside specstar
                          │
                          ├─► KB chatbot (kb/ + api/kb_routes.py, api/kb_chat_routes.py)
                          │     - Ingestor: bytes → SourceDoc (status=indexing) → chunk + embed
                          │       → DocChunk (store + slow index both offloaded via asyncio.to_thread)
                          │     - Embedder Protocol (kb/embedder.py): HashEmbedder (tests),
                          │       LitellmEmbedder (Ollama/hosted; raw vectors stored on DocChunk)
                          │     - Chunker Protocol (kb/chunker.py): FixedTokenChunker
                          │     - Retriever (kb/retriever.py): dense (specstar native vector
                          │       query) + BM25 → RRF → MMR → parent-doc merge; optional
                          │       multi-query / HyDE / rerank when an Llm is wired
                          │     - KB agent = the SAME AgentRunner with a KB AgentToolContext
                          │       (retriever + collection_ids, no sandbox) + the kb_search tool;
                          │       app agents reach it via the ask_knowledge_base tool
                          │
                          └─► specstar (resources/): auto-CRUD for Workspace, AgentConfig,
                                Conversation, and KB Collection / SourceDoc / DocChunk / KbChat
```

Key conventions:

- **The workspace quota is measured live and enforced at the facade (#245/#538).** `filestore.workspace_quota` (0 ⇒ unlimited) bounds ONE workspace's bytes. It is measured against the **live** workspace — warm ⇒ summed from the sandbox's own `walk`, cold ⇒ the durable store's `Sum` aggregate — NOT the durable snapshot alone, because the periodic mirror is deliberately additive (`_writeback(delete=False)`), so a snapshot-based number misses bytes the agent just created AND keeps charging for bytes it deleted (that was #538: "it says full but I deleted everything" and "AI-generated files were free"). Walking on every write would make an N-file folder upload cost N traversals, so `WorkspaceFiles` memoises the walk as a `path -> size` map for `usage_window` (5s, matching `mirror_interval`) and patches it in place on its own writes/deletes — the displayed number trails reality by at most one window. Enforcement lives in `WorkspaceFiles._ensure_headroom`, the chokepoint EVERY write shares (upload, IDE save, copy/move, the agent's `write_file`/`edit_file`, workflows), and one `WorkspaceFull` exception handler in `create_app` turns it into a **507** everywhere — do NOT re-check the quota per route, that's exactly how copy/move stayed ungated for a year. The rule is **growth**, not "already over": a shrink, a same-size replace and every delete are always allowed, so a workspace that went over (the mirror is still ungated — never lose agent work) can always be tidied up. The agent gets a message naming `delete_file` as the remedy, not a traceback. Distinct from `sandbox.max_workspace_bytes` (#345), which is a du cap on the *scratch* dir enforced by recycling.
- **Sandbox is created lazily** by the agent's `exec` tool on first use (grill-me Q10 "a2+" policy). Pure file operations go through FileStore and never spin one up.
- **Workspace files are SHARED-DIR, not affinity-bound (#345).** The local sandbox keys each item to a FIXED dir `{sandbox.root}/{item_id}/root/` on a shared RWX scratch volume, so any replica resolves the same live files by id (`create(spec, sandbox_id=item)` is idempotent; `handle_for_id`/`peek_handle` derive the handle WITHOUT a session; `WorkspaceFiles._warm` PROBES the dir and falls back to the durable FileStore snapshot only when cold). This is what cures "data appears/disappears" on multi-pod — correctness no longer depends on nginx sticky routing (now just a warm-cache optimisation). The FileStore is the durable archive (recycle write-back + blob GC). Recycle (`registry.kill_idle`) rmtrees a shared dir ONLY when GLOBALLY idle — gated by a cross-pod activity heartbeat (`api/sandbox_activity.py`, an opaque `_SandboxActivity` row per item); a pod-locally-idle-but-globally-active item just drops its local session. `sandbox.max_workspace_bytes` adds a du-quota recycle (`registry.enforce_quota`) so a runaway can't fill the scratch vol. `mirror`'s deletion is `_versions`-gated AND now reads the globally-consistent shared dir, so it no longer deletes another pod's live files. Two vols: `sandbox.root` (scratch RWX) vs `filestore.disk_root` (durable). Cross-pod **turn cancel** is still pod-local → #349. Only `kind: local`; http/mock unaffected.
- **HTTP sandbox coherence — share the ADDRESS, not the folder (#366).** `kind: http` is the PRODUCTION backend (the separate `sandbox-host/` service), and its `create` mints a per-pod uuid handle (it does NOT reattach by item id like `kind: local`), so two app pods diverge into two sandboxes for one item → "filetree empty" + `sandbox not found`. Fix: a shared per-item **address store** (`api/sandbox_address.py`, an opaque `_SandboxAddress` row: `item_id → handle_id`, CAS via specstar). `registry._acquire` reads the address, PROBES liveness (`_alive` = `sandbox.exists(handle, "/")`); a dead/absent address triggers a rebuild (`create` → `sync.restore` → `mark_ready`) that is CAS-published (`address.claim`/`swap`, expected=old) — the loser `kill`s its orphan and reads the winner's address, so all pods converge on ONE live sandbox. Readiness is a NEW first-class Sandbox op, `mark_ready`/`is_ready`, backed by an **out-of-workspace** marker (`$root/id/.ready`, a sibling of the `root/` workspace — never in `walk`/the file tree, unforgeable by a user file); the deletion-aware `mirror` honours DELETIONS only while `is_ready` holds BEFORE **and** AFTER its walk (the sandwich), and reap unlinks `.ready` FIRST — so a rebuilt/half-restored/reaped sandbox can never make the mirror wipe the durable snapshot (that was the "temporarily empty vs deleted" ambiguity). The reaper/mirror sweeps are per-item resilient (`kill_idle`/`enforce_quota`/`mirror_warm` swallow one item's error and continue; `kill` treats `SandboxNotFound` as already-reaped). The address store is wired ONLY for `isinstance(sandbox, HttpSandbox)` (registered post-`spec.apply` like `_SandboxActivity`); `kind: local`/mock already converge via the item-keyed shared dir and get no address store. Cross-pod **turn cancel** is still pod-local → #349.
- **Per-sandbox HOME — the carrier launcher never uses a shared /tmp (#393).** The hosted sandbox has NO userns jail (uid+cgroup only), so its `/tmp` is the shared pod `/tmp`. The `python-stack` carrier `launch` used to hardcode `HOME=/tmp`; a user's `pip install --break-system-packages X` can't write the root-owned carrier venv, so pip defaults to `--user` = `$HOME/.local` = `/tmp/.local` — a pod-wide dir every sandbox's carrier python then imports (the cross-sandbox leak). Fix: `_exec_argv` passes `SANDBOX_HOME` to the launcher — unjailed → a per-sandbox `.home` (a `root/` workspace **sibling** in the infra area, so it's never walked/synced and is reaped with the sandbox; `IsolatedProcessSandbox._provision` chowns it to the sandbox uid, 0700); jail → `/tmp` **explicitly** (its `/tmp` is a per-exec ephemeral tmpfs, isolated). The launcher's unset fallback is a PRIVATE `mktemp -d`, NEVER the shared `/tmp` — the default must not be the footgun. `SANDBOX_HOME` survives the `setpriv` uid drop (no `--reset-env`). npm/cargo never leaked (their HOME is already the per-sandbox workspace) and system dirs are uid-blocked, so python was the only leak. `_PYTHON_LAUNCH`/`_LAUNCH` are folded into the `.built` stamp (`_builder_fingerprint`) so a launcher-template edit forces a bundle rebuild (`_source_hash` can't see builder code baked into the bundle — same stale-cache class as #64). Both `local_process.py` + both `isolated_process.py` copies.
- **Per-item OS-user + cgroup isolation (#345, opt-in).** When item dirs are siblings on one shared vol, `sandbox.isolation.enabled` swaps `LocalProcessSandbox` for `IsolatedProcessSandbox` (`sandbox/isolated_process.py`): each item's `exec` drops (via `setpriv`) to a STABLE uid `uid_base + xxhash(item_id) % uid_range` — same on every pod, so ownership + cgroup are coordination-free — under a per-item cgroup v2 slice. NOT the userns jail (`sandbox.isolate`); a different model (no namespaces). `enabled` None=auto (on iff CAP_SETUID + writable cgroup root), True=fail-loud-if-unsupported, False=off; uv-run forces off. Needs the pod to carry CAP_SETUID/SETGID + a delegated cgroup root (`kubernetes/base/deployment.yaml`). Every privileged op is seamed so the module is 100% unit-covered non-root; real enforcement is a root-gated integration test.
- **AgentRunner Protocol** is the swap point between scripted tests and live LLM. Tests use `ScriptedAgentRunner`; production uses `LitellmAgentRunner`. The **KB agent reuses the same runner** — `AgentToolContext` serves both flavours (app workspace agent: sandbox/filestore/sync + file/exec tools; KB agent: retriever + collection_ids + `kb_search`, no sandbox), so the workspace-only fields are optional.
- **Chat turns run through one `ChatTurnEngine`** (`api/turns.py`) shared by app workspace chat + KB chat: per-conversation lock, one cancellable in-flight turn (new message cancels the prior one), the `_drive` pump (CancelledError → `RunCancelled`, other → `RunError`), the SSE `gen()` that reduces events into neutral `TurnMessage`s, and `cancel()`/`forget()`. Each surface only builds its `AgentToolContext` and passes an `on_complete` that maps `TurnMessage` → its model (`Message` / `KbMessage`). `InvestigationRegistry` owns only the **sandbox** lifecycle. Don't reimplement turn/cancel/SSE per surface — extend the engine.
- **SSE event schema** (`api/events.py`) is mirrored in `web/src/events.ts`. Keep them in sync when adding event types. The KB chat streams the same events, and the FE renders both chats with the shared `web/src/components/AgentEntryView.tsx` (reasoning / tool cards / metrics).
- **`kb_search` vs `ask_knowledge_base` — leaf vs consumer-interface; they don't merge (#270).** `kb_search` is the KB agent's own retrieval *leaf* (vector search → numbered `[n]` passages; needs a `retriever` + `collection_ids` in context). Grant it ONLY to agents that *are* the KB agent: the `kb_chat` presets and the `infer_modules` classifier (likewise `search_wiki`/`read_source` only to the wiki maintainer/reader agents). Every *other* app agent gets **`ask_knowledge_base`** instead — it delegates the whole question to a `kb_chat` sub-agent (via the generic `run_subagent` bridge, wired for every app turn in `_send_into` + for workflows; KB-chat turns deliberately omit it so the KB agent can't recurse) and returns a synthesized, cited answer (non-streaming `answer_question`). The point of the sub-agent is **context isolation**: the noisy retrieval (multi-query/HyDE/rerank, raw passages, whole wiki pages) stays in the sub-agent's throwaway context, keeping the consumer's window lean — which matters most for small local models. It still relays the sub-agent's searches + reasoning live into the parent stream via `ctx.on_exec_output` (`answer_question(on_event=…)` + `kb_progress`) so the turn shows progress instead of stalling. **They can't be collapsed into one tool:** `ask_knowledge_base` spawns a KB agent, and that inner agent must search with a *non-delegating* tool (`kb_search`) or it would call `ask_knowledge_base` on itself forever — the leaf is irreducible. So when wiring a new app, list `ask_knowledge_base`, **never** `kb_search`/`search_wiki` (those need a retriever/wiki context the app doesn't have and will fail without). `lookup_glossary` is the exception — a cheap, deterministic, exact-key context-card lookup (no LLM, no retriever), so it's fine to grant any app directly.
- **SourceDoc id is an opaque, slash-free token** (`kb/doc_id.encode_doc_id` = the natural key `{collection}/{user}/{path}` percent-encoded; specstar ids can't hold `/`). **Never parse it** — read `path`/`collection`/`user` from the record + `created_by` meta. `render_document` takes it as a query param (`GET /kb/documents?id=`); the FE treats it opaquely.
- **No-config fallback**: when an item has no attached `AgentConfig`, the turn runs with the **first** config in the store (earliest created), not a bare default — `_resolve_agent_config`.
- **KB ingestion is async + off the loop**: the upload endpoint runs `Ingestor.store` (fast — `SourceDoc` as `status="indexing"`) and schedules `Ingestor.index` (chunk + embed), **both via `asyncio.to_thread`** so the blocking magic-sniff / specstar I/O / embedding HTTP never sits on the event loop; the doc flips to `ready` / `error`. Citations are NOT in the SSE stream — the FE refetches the thread on `done` to get the persisted `[n]` → `Citation`.
- **Job runner ⊥ API (#312)**: the background job coordinators (index / wiki / card-gen / sanity) are built by ONE FastAPI-free composition root, `coordinators.build_coordinators` (+ `build_ingestor` / `resolve_wiki_config`), used by BOTH `create_app` and the standalone worker `python -m workspace_app.worker <jobtype>`. The API gates in-process consuming behind `server.run_consumers` (default `true` = all-in-one for local dev / tests / single pod; set `false` so the API is a **pure producer** — it still `add_model`-registers + `enqueue`s, just doesn't `start_consuming`). A worker pod block-consumes ONE JobType (`consume_until_stopped` = `start_consuming` + park on SIGTERM + `aclose` drain) so each scales under its own k8s HPA (`kubernetes/base/workers.yaml`); a pod-split needs a SHARED backend. The non-queue sweepers (`_idle_killer`/`_mirror_sweeper` = per-pod sandbox reap, plus `_index_sweeper`/`_blob_gc`/`_code_sync`) always stay on the API — they are NOT gated. Build a new coordinator? Add it to `build_coordinators` + the bundle, not inline in `create_app`.
- **Embeddings are computed by us and stored raw** on `DocChunk` (`Vector`, cosine). `KB_EMBED_DIM` must match the embedder's output width; changing it requires re-indexing.
- **specstar singleton vs instance**: always construct a fresh `SpecStar()` instance — never use the module-level `specstar.spec` singleton. This keeps tests isolated.
- **`dict[str, Any]`** (not `dict[str, object]`) for specstar struct fields — `object` breaks JSON-schema generation. Narrow `resource.data` with `assert isinstance(...)` for `ty` (coverage-clean).
- **Backfilling a new index onto existing rows**: specstar extracts `indexed_data` at write time and does NOT auto-backfill — rows written before the index group under `None` and under-count in aggregates. To make them countable, register the model as `Schema("vN").step(None, _reindex_only, source_type=Model)` (the `None` step covers rows written before any `Schema`), then an operator runs the **migrate route** `POST /{model}/migrate/execute` (specstar's `MigrateRouteTemplate` — opt-in, registered globally in `make_spec`) to re-extract their `indexed_data`. `rm.migrate` is the only backfill op; **don't hand-roll a reindex loop** (specstar discussions #365/#366).
- **List/page aggregates must be scoped**: count/sum for a page goes through `exp_aggregate_by(..., query=...)` bounded to the page's ids (or the collection) — never a global group-by just to look up one page (e.g. `doc_cited_for_ids(spec, ids)`, not `doc_cited(spec)`, in `list_documents`). `.contains` is a membership *filter*, not a group-by (it can't produce per-element counts). On an **indexed `list[str]`** field it is EXACT element membership on every backend **as of specstar 0.11.9** — Postgres `@>` / SQLite `json_each`, so `"m4"` does NOT match a card keyed `"m40"` (specstar #378/#362; our #181). The precondition can silently regress: the field must stay in `indexed_fields` AND annotated `list[...]` so specstar registers it as a list field, else SQL `.contains` falls back to substring `LIKE` (and unit tests run on the in-memory backend, which always did element membership, so they won't catch a Postgres-only regression).
- **FE data layer is TanStack Query**: GET-style reads go through `useQuery` (keys in `web/src/api/queryKeys.ts`, one client in `web/src/api/queryClient.ts`); writes are `useMutation` + `invalidateQueries`. SSE stays imperative (`useAgent`/`useKbChat`), but their initial hydration is a `useQuery`. Components/hooks under test need a provider — wrap with `web/src/test/queryWrapper.tsx` (`QueryWrap` / `renderWithQuery`). The signed-in user id is `api.getCurrentUser()` via `useCurrentUser()` (mocked until SSO), not a hardcoded constant.
