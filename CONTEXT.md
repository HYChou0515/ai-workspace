# Domain Context

Names with definitions. Use these terms exactly. Drift to other words ("service", "handler", "manager") in code or commits costs review time.

This file is loaded into context every grilling / architecture session — its job is to keep vocabulary tight, not to repeat what `CLAUDE.md` already covers (architecture map, gates, persistence layer).

## Apps & work items (#89 — RCA is now one App)

The platform is **multi-app**: RCA is one App produced from an in-code template dir; more (e.g. "FOO") can be added by dropping a dir. Clean break — no migration; the old single-`Investigation` model + `agents.workspace_chat` picker are replaced.

- **App** — a self-contained, separately-branded dashboard defined by an in-code dir `apps/<slug>/` = `app.json` (identity + function + agent + layout) + `model.py` (its WorkItem Struct) + `prompts/` + `profiles/`. Routes: `/` = launcher (always shown; lists App cards + a KB link card), `/a/:slug` = an App's home, `/a/:slug/:itemId` = its workspace. **Supersedes `agents.workspace_chat`** (picker is now per-App `app.json.agent.picker`); `agents.presets` + KB usage entries are unchanged.
- **profile** — a named starter-content bundle *inside* an App (`apps/<slug>/profiles/<name>/`): seed files + `_prompt.md` (prompt appendix) + `.skill/` + `_profile.json` `{title, description, suggestions, tools (⊆ app.tools), presets (⊆ app.picker), default_preset}`. Chosen at item-create time. (Was the global "template profile".)
- **WorkItem / WorkItemBase** — the per-App record. Each App hand-writes a `msgspec.Struct` subclass of `WorkItemBase` in `apps/<slug>/model.py`, registered via `add_model` + the App's `INDEXED_FIELDS` (native per-resource indexing; data is **not** mixed across Apps). `WorkItemBase`: **Tier 1** (`title`, `owner` — concrete required) + **Tier 2** (`members`, `topics` — `T | UnsetType`, opt-in by redeclaring as `T`). **Tier 3** = the App's own typed domain fields (RCA's `RcaInvestigation`: `severity`/`status`/`product` as enums/str). No `extra: dict`.
- **item_id** — the cross-cutting key the shared machinery (FileStore, Conversation, sandbox) attaches to = a WorkItem's `resource_id` (already globally unique; never `uid`). Replaces `investigation_id`. `Conversation.item_id: str` is opaque + indexed (one shared Conversation table); deletion cleanup is a per-App **on-delete event_handler** (no declarative Ref/cascade).
- **AppCatalog** — startup-built directory of registered Apps (manifests + profiles) + the per-turn resolve `app (picker / tools ceiling / base prompt) ◇ profile (tools&presets subset / prompt appendix / suggestions / skills) ◇ preset (model / creds)`. `AgentConfigCatalog` shrinks to serve only KB purposes (`kb_chat` / `infer_modules`).
- **function toggles** (app.json) — `workspace` (file IDE + file tools + profile file-seeding), `sandbox` (agent `exec` + package tools), `terminal` (human shell pane; requires `sandbox`). A `tools[]` ↔ toggle incoherence (e.g. `exec` listed but `sandbox:false`) is a **startup hard error**.
- **layout** (app.json) — per-surface field placement: `breadcrumb` / `statusbar` / `list` / `form` each list (ordered) which fields show; `default_tabs` lists the files the workspace opens on entry (filtered to those that exist; replaces the old hardcoded `designViews`). app.json is a display-only overlay (`layout` + optional `labels` + `field_styles`). Item display nouns come from `item.{noun, noun_plural, create_label}`.
- **field schema** — the per-field `{name, label, kind: select|text, options?}` the FE needs to render + inline-edit a WorkItem's domain fields. Projected by the backend from the model's OpenAPI schema (a field with an `enum` → `select` + its values as `options`; else `text`) and folded into the `GET /apps/{slug}` manifest response as `fields`. Mirrors specstar **autocrud**'s `ResourceField` (its codegen bakes the same shape from `/openapi.json` at generate time) — we instead serve a lean subset at runtime, so no FE codegen. The model is the single source of enum options; never restate them on the FE.
- **field_styles** (app.json) — optional overlay mapping an enum field's options to semantic tone tokens (`danger`/`warn`/`ok`/`muted`/`accent` → existing CSS vars), e.g. `{severity: {P0: "danger", …}}`. The `select` renderer styles its chip from this; absent → neutral. Theme-aware (no raw hex) so dark mode doesn't break. RCA's `SeverityChip`/`StatusChip` colours survive here as DATA, not code.
- **lifecycle** (app.json) — optional `{status_field, closing_states}` declaring an App's close/resolve workflow. The shell shows a Close affordance only when present; it offers `closing_states` (a subset of the status field's enum) and tears the sandbox down generically. No lifecycle → no Close. (specstar's CRUD `PUT /{route}/{id}` also takes a `change_status` param the close path can use.)
- **WorkspaceShell / DomainFields** — the workspace screen is ONE generic shell (was the RCA-specific `InvestigationShell`); it takes a generic `WorkItem` + the `AppManifest` + the field schema, and renders domain fields through a `DomainFields` renderer driven by `layout` × field schema. The only per-field-TYPE code is a small kind→renderer registry (`text`, `select`) — a real seam with ≥2 adapters; per-App variation is pure data (app.json + model.py), so adding an App needs no shell edit. Reads/writes go through specstar's CRUD routes (`GET/PUT /{resource_route}/{id}`, the same client autocrud generates); inline-edit PUTs the whole item with one field changed.

## Agents

- **Preset** — a named LLM recipe in `agents.presets`. `{model, prompt_file?, suggestions, allowed_tools?, env, sandbox_image, idle_timeout_seconds, llm: {base_url, api_key}}`. Reusable across multiple callers (RCA picker, KB chat, infer_modules sub-agent, kb.retrieval_llm). Bundled presets: `qwen3-local`, `claude-opus`, `openai-mini`, `kb-default` (KB sub-agent), `infer-modules-default` (module classifier sub-agent), `kb-retrieval` (LLM-only, no prompt/tools — for the retriever's multi-query / HyDE / rerank). `prompt_file` is optional: LLM-only presets like `kb-retrieval` legitimately omit it; agent-style callers enforce non-empty at catalog build.
- **Usage entry** — one dict that references a `Preset` by name + an optional subset of preset fields as inline overrides. Lives in `agents.workspace_chat[]` (RCA picker), `agents.kb_chat[]` (KB chat picker), `agents.infer_modules[]` (module-classifier sub-agent), and `kb.retrieval_llm` (single entry, not a list; can be `null` to disable). The catalog merges `usage entry ◇ preset ◇ runner defaults` into an `AgentConfig` at build time. The `preset` field is REQUIRED — operators who want a fully inline config write a new preset and reference it.
- **AgentConfig** — one entry the FE picker offers. `{name, model, system_prompt, suggestions, allowed_tools, env, sandbox_image, idle_timeout_seconds}`. Pure data (msgspec.Struct). Not a specstar resource — lives in deploy config (`Settings.agent_configs` / `config.yaml`), not in the DB. Built by the catalog from a `Preset` + a usage entry.
- **AgentConfigCatalog** — the runner-time directory of available `AgentConfig`s + the resolver that picks one for a given investigation. Owns: `list()` / `get(name)` / `default()` / `resolve(attached_name, template_profile)`. The resolve cascade is `attached → template → default` plus the template-appendix composition onto `system_prompt`. Lives in `agent/config_catalog.py`. The FE's `/agent-configs` route reads from one of these; `create_investigation` auto-attach reads `default()` from it; per-turn agent selection reads `resolve(...)` from it. **The catalog replaces** the `LitellmAgentRunner.list_configs / get_config` Protocol methods and the four `app.py` closures (`_runner_configs`, `_runner_get_config`, `_first_agent_config`, `_resolve_agent_config`) that previously did the same work in pieces.
- **AgentToolContext** — the per-turn context the agent's tools receive (sandbox, filestore, ask_kb bridge, etc.). Two flavours (RCA / KB) share one dataclass; "wrong flavour" is signalled by optional fields being `None`. Carries the resolved `AgentConfig` per turn.
- **AgentRunner** — the Protocol that drives one turn. Post-catalog refactor, knows nothing about the picker — just runs a turn given a context. Implementations: `LitellmAgentRunner` (prod), `ScriptedAgentRunner` (tests).

## KB retrieval enhancements

- **Enhancements** — per-search override knobs for the retriever's LLM-driven enhancements. Dataclass `{expand: int | None, hyde: int | None, rerank: bool | None}` in `kb/retriever.py`. `expand` = number of alternative query phrasings (`0` = off); `hyde` = number of hypothetical-document probes (`0` = off); `rerank` = LLM rerank pass after fusion. `None` per field means "inherit from the operator default for this knob".
- **EnhancementSettings** — operator-level defaults + ceilings (`kb.retrieval.enhancements` in `config.yaml`). Each knob is `{default, max}` (`EnhancementInt` for expand/hyde, `EnhancementBool` for rerank). Bundled defaults are **light**: `expand=1, hyde=0, rerank=true` — sized so a fresh deploy isn't slow.
- **Resolution cascade** — LLM tool args (`kb_search(query, expand?, hyde?, rerank?)`) win over Python caller (`Retriever.search(..., enhancements=Enhancements(...))` and the `AgentToolContext.kb_enhancements` set by the API layer) over operator `default`. The operator's `max` clamps the result — int knobs are floored at 0 and capped at `max`; bool's effective value is `raw AND max`, so `max=False` is a hard kill switch.

## Tool I/O — handling streaming aggregator merges

The LiteLLM streaming layer occasionally merges parallel `tool_call.arguments` strings into one concat-JSON string. This is a framework-level event, not small-model-specific (small models trip it more often, big models can too).

- **args_recovery** (`agent/args_recovery.py`) — the FunctionTool wrap that catches the merge at invoke time. Peels the first JSON object, raises `ValueError` with "Extra data" in the message. The runner's `diagnose_error` routes that to the "send ONE tool_call per response" retry hint; the run restarts on a clean conversation.
- `_map_event` exception branch (`api/litellm_runner.py`) — when the streaming-aggregator concat reaches the event reducer, `args_obj` becomes `{}` (not a sentinel). The sentinel pattern is gone.

Anything that touches `arguments` strings (decode at events, raise at invoke, project to history) goes through one of those two places. There is no shared "tool I/O codec" module — each touchpoint is shallow enough on its own.

## API layer decomposition (#54 — `create_app` is a composition root)

`api/app.py` used to carry the whole workspace API (a 3200-line `create_app`). It's now a composition root: it builds the service bundle, wires the deep modules below, and registers the route groups. Each module owns one concern behind a small interface, so a route group or a turn surface can be read, tested, and changed in one place.

- **item locator** (`api/locator.py` — `ItemLocator`) — resolve an item's identity (owning App slug / profile / title / `AgentConfig`) and its conversations (default chat, engine key, chat validation) from an opaque `item_id`. Wraps `apps.resolve.find_work_item`; the cross-cutting vocabulary every `/a/{slug}/items/{item_id}` route crosses. The `_item_slug` / `_require_item` / `_conversation_for` closures collapsed into it.
- **turn-context builder** (`api/turn_context.py` — `TurnContextBuilder`) — assemble an RCA turn's `AgentToolContext` once. `build_chat_turn` = the interactive surface (composer effort/enhancements, collection scope/tiers, acting user + speaker + directory); `build_workflow_turn` = the lean background-node surface (the shared core only). Replaces the duplicated 37-field constructions in `_send_into` + the workflow driver, which had drifted apart.
- **workflow execution adapter** (`api/workflow_exec.py` — `WorkflowExecutor`) — bind the live services into a workflow run's callbacks (agent turn / sandbox / ingest / card upsert+find / landed-check + the orchestrator's upload-dir / wire-handle / release / notify-failure). The `_wf_*` family, collapsed; the seam `_wf_wire_handle` already named.
- **sub-agent bridge** (`api/subagent_bridge.py` — `SubagentBridge`) — the ONE bridge `ask_knowledge_base` / `infer_modules` reach their KB sub-agent through (resolve purpose config → scope collections → drive `answer_question` on the wiki-aware vs base runner → relay live work + log citations). Shared by the turn glue and the `WorkflowExecutor`. (Was the `_run_subagent` closure.)
- **route groups** — `register_*_routes(router, …)` modules following the established KB/health/notifications pattern: `meta_routes` (user/app catalog + activity/monitor), `item_routes` (work-item create/close), `file_routes` (IDE files / notebooks / exec), `capability_routes` (workflow HTTP capabilities + export), `workflow_routes` (profiles + run lifecycle), `chat_routes` (workspace + multi-chat; the turn-driving `send_into` is injected, not owned). `schemas.py` holds the request/response bodies; `rca_messages` / `timeutil` / `promote` / `chat_info` hold the pure helpers they share.

`_send_into` (append user message → build ctx → enqueue the turn) stays in `create_app` and is injected into `chat_routes` — it's the turn glue that binds the locator, the turn-context builder, and the sub-agent bridge, so it lives next to the composition root rather than behind its own seam.
