# #624 — context 上限:偵測、處理、告知

**問題一句話**:我們從來不知道模型能吃多少,卻用兩個憑空的常數決定要忘記多少,而且忘記的時候不告訴任何人。

使用者長期回報的「問一問 AI 就忘記前面」,根因在此。

---

## 1. 現狀(全部已查證,附證據)

### 1.1 偵測:不存在

`grep -r "v1/models|get_model_info|max_input_tokens|max_model_len" src/workspace_app/` → **零個命中**。

我們從未詢問任何 endpoint「你能吃多少」。

### 1.2 兩個憑空常數決定記憶

`api/turns.py::history_items` 實際做的兩件事:

```python
if max_messages:  msgs = msgs[-max_messages:]        # config: 40
if max_tokens:    msgs = _fit_token_budget(msgs, …)  # config: 24000
```

兩個值寫死在 `config/schema.py::HistorySettings`,與模型真實上限無關。

### 1.3 系統提示與工具規格完全不在預算內

`litellm_runner.py:271` 直接送 `config.system_prompt`。實測一個 RCA item 的系統提示為 **74,222 字元**,外加 31 個工具 schema。預算只裁歷史,看不到這塊。

### 1.4 估算器用錯:同 repo 有兩個,聊天用了對中文錯的那個

| 位置 | 算法 | 同一段中文對話 |
|---|---|---|
| `api/turns.py::_est_tokens`(**聊天在用**) | `chars // 4` | 2,435 |
| `kb/tokens.py::count_tokens`(#88 為繁中語料而寫) | CJK 1 字 1 token + 其餘 /4 | 7,446 |
| **模型實際回報** | — | **8,755** |

聊天路徑**低估 3.6 倍**。

### 1.5 裁切完全靜默

`_fit_token_budget` 結尾只有 `return kept` — 不回報丟了幾則、不記 log、不發事件。**資訊在源頭就被扔掉**,不是「有但沒顯示」。

### 1.6 沒有任何摘要 / 壓縮功能

全 repo 的 `summariz*` 命中都在 KB(code wiki 大綱、VLM 描述、merge、eval),沒有一個為了把對話塞進 context。

### 1.7 400 的處理是反效果

- `_should_retry` **只看**「有沒有吐字」與「試了幾次」,**不看錯誤類型** → 同一個過長 prompt 重送 **3 次**
- `diagnose_error` 落到 catch-all,回饋一句 "The previous attempt failed… Try again." **附加到那個已經太長的 prompt 上,使它更長**
- 400 不在 `failover/retry.py:46` 的 transient 集合 `{408,409,429,500,502,503,504}` → 不換模型(這點正確)
- 使用者最後看到 `RunError("giving up after 3 attempts: …")` 的原始英文錯誤,沒有任何可行動指引
- 使用者訊息在 turn 前已持久化,失敗又寫入一則 `role="error"` → **歷史更長 → 下一輪更容易再失敗**

### 1.8 文件宣告了一個從未驗證的前提

`configs/config.example.yaml`:

> *「Default sized for the bundled local qwen3 (~32K ctx); raise it for big-context hosted models.」*

而 `num_ctx` / `OLLAMA_CONTEXT_LENGTH` 在 `configs/`、`docs/`、`kubernetes/` **grep 零命中**。照文件裝起來的部署,拿到的是一個每輪被截斷的 agent,而文件讓人以為一切正常。

## 2. 已量測的衝擊

### 2.1 我們自己砍掉多少(provider 無關,production 一定發生)

每個工作回合 = 4 則訊息(問 → agent 說明 → 工具輸出 → 結論):

| 情境 | 從第幾回合開始丟 | 穩定後 AI 看得到 |
|---|---|---|
| 輕量(工具輸出 1.2K 字元) | 第 **11** 回合 | 最後 40 則 ≈ **10 個回合** |
| 常見(工具輸出 30K 字元 = `exec` 上限) | 第 **4** 回合 | 最後 13 則 ≈ **3 個回合** |

第 60 回合時第二種情境:240 則只送出 13 則,**丟掉 95%**。

**且與 vLLM 開多大無關** — 這是我們在送出前自己套的天花板,對方的 context size 目前是一個從未被用到的數字。

### 2.2 Provider 失敗模式不同(dev 實測 / production 待驗)

| 服務層 | 超長時 | 證據 |
|---|---|---|
| **Ollama** | **靜默截斷**,不報錯 | 實測 `prompt_eval_count` = 3,983 / 8,755;同對話 `num_ctx=40960` 則完整看到並答對 |
| **vLLM** | raise 驗證錯誤(`invalid_request_error`) | 官方文件 `_token_len_check`;**未在你們機器實測** |

Ollama 被截斷時**不會說「我不知道」,會編一個看似合理的答案**(實測:問開頭的專案代號,答「TSMC-2023-001」,實際是 ORCHID-7)。

### 2.3 上限的可得性

| 來源 | 結果 |
|---|---|
| `litellm.get_model_info("gpt-4o")` | 128,000 ✅(**真的在內建表裡**) |
| `litellm.get_model_info("ollama/qwen3:14b")` | 40,960 —— **但不是查表**(見下) |
| **`litellm.get_model_info("openai/qwen3-14b")`** | **查不到 ❌**(= production 的形狀) |
| Ollama 的 OpenAI 相容 `/v1/models` | 只有 `{id, object, created, owned_by}`,**無長度欄位** |

→ 對自架 vLLM,內建表無效;`max_model_len` 是 vLLM 的擴充,非 OpenAI 標準。

**更正(實作後查證):** `ollama/qwen3:14b` **不在** litellm 的靜態表裡(`model_prices_and_context_window_backup.json` 查無此鍵)。它會回 40,960,是因為 litellm 去問了**本機正在跑的 Ollama daemon**。所以「catalog 這一階能覆蓋 `ollama/*`」只在該機器已經跑著那個模型時成立——這是完全不同的一句話,而且**單元測試守不住**(這條斷言在我的機器上綠、在 CI 紅)。真正覆蓋自架模型的是 `learned` 與 `probe` 兩階。

> **判準**:凡是「某個表查得到 X」的量測,要在**沒有那個服務在跑**的環境重測一次,否則量到的是自己的機器,不是那個表。

### 2.4 反推所需的儀器早已存在但未被使用

`litellm_runner.py::_exact_usage` 已在收 `usage.input_tokens`(模型實際吃進去的 token),並已流入 `AgentMetrics`。**但從未與我們的估算做比對** — 只拿去顯示 UI 的 ↑ 數字。

## 3. 鎖定的決策

| 決策 | 內容 |
|---|---|
| 預設行為 | **不預先裁切**。上限多大就用多大,不自斷腳筋 |
| 上限單位 | 可設定;能拿到真值就用 token 真值,否則用估算 + 餘裕 |
| 超長處理 | 400 → **套用同一個縮減演算法後重試**(而非現在的重送同一份 3 次)。錯誤有寫明上限時就用那個預算跑;沒寫明才盲砍一半 |
| 縮減告知 | **一定要說一聲**。轉折點在對話裡留可見標記,**文字由縮減演算法自己產生**(「N 則不會被讀到」對「摺疊」那一段是錯的——它一則都沒丟)|
| **縮減策略** | 「塞不下時優先放棄什麼」是產品決定,不該焊在請求路徑裡 → 放在 `IContextReducer` 介面後面。但**只實作一種**(見下),不做選單、不加設定旋鈕 |
| 縮減演算法 | **三段階梯,依「代價由低到高」排序**,每一段只因為前一段不夠才執行:①摺疊舊的大型工具輸出 ②從中段丟(**保住第一則使用者訊息**)③連任務都放不下,只留當前輪 |
| 摘要 / 壓縮 | v1 不做,但它是**這個介面的下一個實作**——介面的存在是為了它,不是為了今天給選擇 |
| `truncate_prompt_tokens` | **不用**。那是把靜默截斷外包給 vLLM,正是要消滅的東西 |
| config 與現實衝突 | config 決定行為;但當 400 證明 config 是錯的,**以現實為準 + 警告 operator** |

## 4. 分支樹(每條都要有活路)

```python
# ── 送出前 ───────────────────────────────────────────
limit = (config_limit                  # ① operator 說了算
         or learned_limit(model, url)  # ② 之前從流量學到的
         or litellm_limit(model)       # ③ 內建表
         or None)                      # ④ 還不知道

if limit is None:
    送出全部                           # 預設不卡 → 靠回應學
else:
    est = count_tokens_cjk(訊息)
    if est + 系統提示 + 工具 + 回覆保留 <= limit:
        送出全部
    else:
        套用縮減演算法(三段階梯)+ 用它自己的說法說一聲

# ── 收到回應後(所有分支都做)─────────────────────────
P = usage.prompt_tokens                # 已經在收了
if P is not None and P << est:
    learned_limit = P                  # 它偷偷截斷了,P ≈ 真實視窗
    警告 + 在對話裡說一聲
elif P is not None:
    校準估算器誤差(選配)

# ── 收到 400 ─────────────────────────────────────────
if 是長度類 400:
    learned_limit = parse(錯誤訊息)     # vLLM 訊息內含上限
    if 只剩 1 則訊息:
        fail loud:「這則訊息本身就超過上限」+ 指出是哪則
    else:
        套用同一個縮減策略(以錯誤裡寫明的上限為預算)→ 重試
        錯誤沒寫明上限 → 盲砍一半(沒有預算可瞄準時唯一能做的)
else:
    直接失敗,不重試
```

**每個 leaf 的活路:**

| 分支 | 活路 |
|---|---|
| 不知道上限 | 照送 → 從 usage 或 400 學到 → 下一輪就知道 |
| 估算器不準(15%) | 只在明確超過時裁 + 留餘裕;寧可被拒也不預先亂剪 |
| 被靜默截斷 | usage 反推抓到 → 學到上限 + 告知,不會一直錯下去 |
| 砍半砍到底 | 剩 1 則仍爆 → fail loud 指出是哪一則,不無限迴圈 |
| 非長度 400 | 立刻失敗,不浪費 3 次重試 |
| **config 設錯(設太大)** | 400 證明後**以現實為準 + 警告**,逃生門不會反鎖 |

## 5. Phases

> 順序原則:**不依賴 operator 提供資訊的先做**;`/tokenize` 探測晚做,因為沒有它整套仍然可用。
> **P7 是實作後補上的**(見 §9.4)—— 它修正了 P2/P4 與 §4 分支樹裡一個沒被察覺的假設。

### P1 — 純函式:上限解析器 + 準確估算(零行為改變)

**Goal.** `resolve_limit()`(四層階梯,回傳值 + 來源標記)與換用 CJK-aware 計數,**都以純函式落地並測試,先不接進裁切路徑**。

- 分開落地的理由:單獨換估算器會讓同一段中文算出 3.6 倍 token → 24,000 的預算提早觸頂 → **健忘變更嚴重**。估算器與預算推導必須同時生效(P2)。
- 測試:四層階梯各自命中/落空;CJK 估算對照實測值。

### P2 — 送出前分支:預設不卡 + 裁切要說一聲

**Goal.** 兩個憑空常數退休,改由 `limit` 推導;`limit` 未知時不裁切;真的裁切時在對話留可見標記。

- `max_messages` 不再當守門員(記憶由 token 決定,不是「第幾則」)
- 預算 = `limit − 系統提示 − 工具 schema − 回覆保留 − 餘裕`
- 標記沿用 #613 的 `role=` 標記機制(FE 會渲染、`history_items` 天生不收非 user/assistant/tool 的 role)
- 只在**轉折點**響一次,不是每輪都響(否則第二輪就變壁紙)
- ~~裁切要**整塊裁**,不要每輪掉一則 — 前綴每輪變動會讓 vLLM 的 prefix cache 全失效~~
  **由 P7 的摺疊取代,而且更好。** 「整塊裁」是為了讓前綴多輪穩定;但摺疊是**逐則**、**與上下文無關**的變換——同一則工具輸出無論對話長成什麼樣,都摺成同一段文字。於是前綴天生逐字不變,只有在「這輪起需要摺疊」那個轉折點失效一次,而不是每輪失效。滑動視窗做不到這件事:它每長一輪就往前挪一則。
  註:此處只保證常見路徑(階梯①)。階梯②會移動保留的中段,前綴隨之改變——但它很少被觸發,而且到那個地步時「送得出去」本來就比「快取命中」重要。

### P3 — 事後偵測:usage 反推靜默截斷

**Goal.** 每輪比對「我們送的估算 E」與「provider 回報的 `usage.prompt_tokens` P」;`P << E` 判定為靜默截斷 → 學到上限、警告、在對話裡說一聲。

- 這是讓 Ollama 那類靜默截斷**現形**的唯一機制
- 需要防誤判(短 prompt 本來就 P 小):要求差距顯著且可重複觀測才下結論
- 學到的值快取於 `(model, base_url)`,可被新觀測推翻(不是永久)

### P4 — 400 的正確處理

**Goal.** 分類 400、從錯誤訊息學上限、砍半重試、觸底 fail loud、非長度 400 不重試。

- 改 `_should_retry`:目前不看錯誤類型,是「重送同一份 3 次」的根源
- 改 `diagnose_error`:長度類要有自己的分支,不能落到那句會讓 prompt 變更長的 catch-all
- config 設太大時以現實為準 + 警告

### P5 — `/tokenize` 探測

**Goal.** 能拿到真實 token 數時就用真值,取代估算。

- 每個 `(model, base_url)` 問一次並**快取包含「沉默」**;404 / 逾時 / 非 vLLM / 任何失敗 → 靜靜退回 P1–P4(不影響可用性)
- 不再阻擋:防禦性實作 + 8 種拿不到答案的路徑都有測試。但 §7 的 curl 仍需驗證真實回傳格式
- 需實測每輪多一次 HTTP 往返的延遲代價

### P6 — 文件與部署指引

**Goal.** 讓文件不再宣告未經驗證的前提。

- `configs/config.example.yaml`:移除「~32K ctx」這個假設句;寫明新旋鈕;寫明 Ollama 預設 4096 與 `OLLAMA_CONTEXT_LENGTH`(含 VRAM 代價)
- 各 provider 的失敗模式對照表(誰報錯、誰靜默截斷)

### P7 — 縮減策略:把「放棄什麼」從演算法變成政策

> **這個 phase 是後補的。** 原本的 plan 從 §4 的分支樹到 P2/P4 全都假設「縮減 = 丟舊訊息」,而那是一個產品決定被寫成演算法。實作出來後才被指出:它在同樣預算下**先犧牲使用者最初那句需求**(任務本身),卻保住一坨沒人會重讀的工具傾印。

**Goal.** `IContextReducer` 擁有這個決定;請求路徑不得假設任何一種。

- 介面 / 實作分模組(`context_reduce.py` / `context_reducers.py`),`abc.ABC` 命名 `I<Name>`
- **只有一種實作,沒有設定旋鈕。** 中途曾做成四種可選(`drop-oldest` / `elide-tool-output` / `keep-task` / 合成版)——那是把「不做決定」偽裝成「給彈性」:三種永遠不會在任何地方跑,也就等著爛掉。介面的正當性來自**下一個**實作(摘要),不是來自今天的選單;只有一種合法值的旋鈕只能被設錯,所以不留
- **階梯的順序就是它的全部理由**:每一段的代價嚴格高於前一段,所以便宜的犧牲一定先用盡,才會動到貴的
  - ①**摺疊工具輸出**——最便宜。工具輸出是工作階段的大宗(一個 30 KB `exec` ≈ 二十個回合的對話),而且最不值得逐字重讀。**實測:六回合的對話 12,153 → 279 tokens,一則訊息都沒少**。所以它排第一——它幾乎總是就夠了,後兩段很少被觸發
  - ②**從中段丟**——開始失去對話,但不失去方向
  - ③**連任務都丟**——只在「最初需求 + 當前輪」都塞不下時。最新一則永遠留著:沒有自己上下文的一輪會產生語無倫次的回覆,比健忘更糟
- **兩條路徑共用同一個演算法**:送出前的預算縮減,與 400 之後的重試。同一個產品決定不該有兩套答案(第一版就是,而且兩套不同)
- 縮減的**說明文字由演算法自己產生**——「N 則訊息不會被讀到」對「摺疊」那一段是錯的(它一則都沒丟)
- 必須對**兩種訊息形狀**都有效(持久化的 `Message` 物件、SDK 的 dict)。只認一種不會報錯,而是**靜默無作用**

## 6. 非目標

- **對話摘要 / 壓縮** — v2。撞到天花板時誠實告知並建議開新對話即可;它是 `IContextReducer` 的下一個實作,加一個 class,不動請求路徑
- **vLLM `truncate_prompt_tokens`** — 刻意不用(靜默截斷)
- **調整 `exec` 輸出上限**(30,000 字元)— 另一個議題,雖然它是吃掉預算的主因
- **縮減系統提示**(#480 把全部工具 schema 寫進提示,佔 74,222 字元)— 值得另案評估,不在此範圍

## 8. 風險

| 風險 | 緩解 |
|---|---|
| 估算器仍有 15% 誤差 | 保留餘裕;真正的準確度靠 P3 的 usage 反推校準 / P5 的真值 |
| 學到錯誤的上限 | 需顯著差距 + 可重複觀測才下結論;可被 config 覆寫、可被新觀測推翻 |
| 裁切破壞 prefix cache | 整塊裁,讓前綴多輪穩定 |
| unknown 期間第一輪未校準 | 第一輪就會偵測到並告知,不會持續錯下去 |
| P2 上線後使用者開始看到裁切通知 | **這是預期的** — 它讓一直存在但被隱藏的天花板變得可見 |

## 9. 實作後的教訓(對抗式複查 + 使用者複審)

這一節記錄「plan 對、code 不對」的具體案例,因為它們不是打字錯誤,而是可重複發生的模式。

### 9.1 綠燈的純函式不等於功能

第一版把 P3 / P4 / P5 的邏輯寫成純函式並全部測綠,就宣告完成。**變異測試證明它們全是死程式碼**:把 `LimitLearner`、`detect_truncation`、`halve_history`、`parse_limit_from_error`、`_tools_tokens` 整組刪掉,**60 個測試照樣通過**——只有它們自己的測試檔知道它們存在。

> **判準**:每個新函式都要能回答「revert 掉它的呼叫端,哪一條測試會紅?」答不出來就是還沒接線。

### 9.2 §4 的樹是對的,是實作漏了一條活路

plan 明寫「400 → 砍一半重試」,但實作只做了「不重試」。結果在 production 形狀下(自架模型、catalog 查不到、未設 `context_limit`):歷史無限成長 → 400 → 失敗被持久化 → 歷史更長 → 又 400 → **對話永久壞死**。而改動前的 `max_messages=40` 本來擋著這條路。

> **判準**:分支樹的每個 leaf 都要有一條測試直接驅動它,而不是只在文件裡宣稱有活路。

### 9.3 只認一種資料形狀的函式不會報錯,只會靜默失效

`estimate_messages` 只認物件、不認 dict,對 SDK history 一律回 0 → 所有東西都「裝得下」→ 縮減政策整個變成 no-op → 呼叫端退回它為緊急情況保留的盲砍規則。**於是一個會保住任務的政策執行了、回報成功、卻什麼都沒做**,而任務照樣被丟掉。

> **判準**:凡是接受「訊息」的函式,兩種形狀都要有測試。

### 9.4 提供選擇 ≠ 不做選擇

seam 蓋好之後,一度把 `drop-oldest`(現任)留作預設,理由是「這是產品判斷,交給使用者」。那等於**把這個 issue 要修的行為原封不動出貨**,只是多附一張選單。

> **判準**:預設值要用證據選(這裡:工具輸出佔量最大且最不值得逐字重讀;第一則訊息是任務本身),並讓其他選項一行可換。「留給使用者決定」不能當成不決定的藉口。

### 9.5 決定做完之後,不要留下沒人會選的選項

§9.4 說「提供選擇不等於不做選擇」。它的鏡像同樣會咬人:決定做完之後,**把落選的選項留在程式碼裡當「彈性」,是把同一個逃避換個位置**。四種政策裡有三種永遠不會在任何部署跑起來,也就沒有任何東西會發現它們何時壞掉;而一個只有一種合法值的設定旋鈕,唯一的作用是被設錯。

介面留著是對的——**但它的正當性來自下一個實作(摘要),不是來自今天的選單**。

> **判準**:每個抽象都要能指名「誰是第二個實作」。指不出來,就先寫成一種做法。

### 9.6 「共用同一個演算法」不包含算術,就等於沒共用

P7 要求送出前與 400 之後**用同一個縮減演算法**。程式碼照做了,但兩邊餵給它的**預算**是不同的:

| 路徑 | 預算 |
|---|---|
| 送出前 | `limit×0.9 − 系統提示 − 工具 schema − 回覆保留` |
| 400 之後 | `stated`(**原封不動**) |

400 訊息報的是模型的**總**上限,可是那個請求同時載著 74,222 字元的系統提示與 31 個工具 schema。拿總上限去量歷史,會判定「塞得下」→ 演算法原封不動回傳 → 退回盲砍一半 → 再被拒。實測 log:`6 → 3 → 2 → 1`,三次白費的 round-trip,最後吐「這則訊息太長」,把系統提示花掉的額度算在使用者頭上。

同一段程式碼還有第二個單位錯誤:「有沒有進度」是用**訊息則數**判斷的(`len(new) < len(old)`)。但階梯第①段摺疊工具輸出時**一則都不丟**——它的進度在體積。於是一次已經釋出 97% 預算的摺疊會被判定「沒進度」而丟棄,改用盲砍一半:**為了留住一坨資料傾印而丟掉任務**,正是這個 issue 要消滅的行為,在重試路徑上原地復活。

> **判準**:說兩條路徑「共用同一個決定」時,要把**輸入**一起共用——同一個演算法配兩套預算,是兩個決定。而衡量進度的單位,必須是該演算法真正會改變的那個量。

### 9.7 變異測試的紅燈,有時是逾時

用來驗收 §9.1 的變異測試(拆掉 `ctx.history = _shrink_history(...)`)沒有讓任何測試變紅——它讓測試**跑不完**(577 秒仍在轉)。因為 400 的重試迴圈刻意繞過 `_should_retry` 的次數上限,它的終止**完全**依賴那一行每輪都讓歷史變小。

那一行確實有測試在驅動(所以變異有效),但回歸在 CI 會表現成逾時而不是失敗訊息。

> **判準**:凡是靠「每輪都有進度」終止的迴圈,進度保證要寫在迴圈裡看得見的地方,不能只藏在被呼叫函式的內部。

### 9.8 學到了、但沒有人花用

變異測試把 `turn_context` 的 `learned=self._learned_limit(...)` 改成 `learned=None`,**86 個測試全綠**。也就是說 P3 辛苦從流量學到的上限,沒有任何測試驗證它會改變下一輪送出的量。

這正是 P3 的全部意義:在自架模型上 catalog 查不到、config 也沒設,那條 `learned` 是**唯一**會回答的階梯。它若沒被消費,runner 就永遠在重新學同一個數字,而預算永遠停在「不知道」。

> **判準**:凡是「A 學到、B 使用」的交接,測試要**跨過交接點**斷言 B 的行為改變,而不是分別測 A 學得對、B 算得對。

### 9.9 說了「一定要說一聲」,只有三分之一的表面在說

§3 把縮減告知寫成**無條件**。實作只有 app chat 做到:

| 表面 | 縮減時 |
|---|---|
| app chat | 留下 `role="notice"` ✅ |
| **KB chat** | 完全靜默 —— 而它是最會撞天花板的(檢索段落、整頁 wiki 都往這條 thread 塞) |
| **workflow turn** | `_common` 算好了說明,呼叫端直接丟掉 |

「背景跑的沒人在看」不是略過的理由,而是**要持久化**的理由——事後打開那條對話的人,才是需要知道的人。三處現在共用 `turns` 裡同一份措辭與「只說一次」規則。

> **判準**:寫下「一律要 X」時,先數清楚有幾個入口點會走到 X,並逐一列表——這正是[開關要逐入口點問範圍]的同一條規矩,用在行為上而不只是旋鈕上。

### 9.10 文件會重新長出謊言

§1.8 指控 `config.example.yaml` 宣告了一個從未驗證的前提。修好之後,對抗式複查把 P3/P5 接上了線,而同一份文件裡「**尚未接線(請勿依賴)**」那段沒有跟著改——於是它開始叫 operator 手動設 `context_limit`,理由是一個已經不存在的缺口。同一個檔案、同一種錯,相隔幾天。

> **判準**:凡是文件裡描述「目前還沒有 X」的句子,都要當成會過期的斷言;接上 X 的那個 commit,必須同時刪掉那句話。
