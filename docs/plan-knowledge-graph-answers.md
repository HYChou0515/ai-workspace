## 1 · 為什麼要有這個

使用者定調(2026-07-23/24),圖存在的理由是回答兩類問題:

1. **定義 / 設定題** — 「n5 tv0 por recipe 是什麼」
   期待的答案**不是字典定義**,而是一張檔案卡:名稱、參數、當時的 quality
   review 報告、對應的 slide。
2. **因果題 + 前例** — 「n3 tv1 ox broken 的原因是?有沒有過去經驗可以對照?」
   前半是指向這個失效的因果邊(附原句);後半是同類事件在別的 node / 時期的
   紀錄。

語料特性:**大量簡報,裡面是表格、圖示、箭頭**。這正是向量檢索最弱的地方 ——
答案散在很多張投影片上,要靠「拼」而不是「找一段話」。

> ⚠️ **內部術語極多,任何行話都不准進 prompt**。使用者給的例句只是形狀示範,
> 機制必須通用,不得對範例 overfit。

### 答案的形狀:檔案卡

兩類問題共用同一種答案 — **把一個東西的檔案卡拼出來,每一行附出處**:

```
名稱(含別名) · 類別
文件怎麼說它     良率 = 98.7% (Q3)   [deck-A]
                 POR recipe = PPOOIXUX  [deck-A]
誰用了它         回焊爐 · POR recipe   [deck-A]
它和誰有關       回焊爐 —造成→ 空洞「回焊爐溫度過高造成空洞」[deck-A]
哪些文件提到它   deck-A / deck-B / deck-R
```

## 2 · 業界調研(四路平行,2026-07-24)

完整對照表與出處見 [#628 的調研留言](https://github.com/HYChou0515/ai-workspace/issues/628#issuecomment-5064004061)。結論三句話:

- **沒有任何一家用「數字 vs 文字」當分界。** 一家都沒有。LangChain 甚至強制
  `value: str`。我們原本是唯一一個。
- **真正的分界是「你要不要談論它 / 跨它查詢」。** RDF 是硬法則(literal 不能
  當主語);Neo4j 官方原話:「要按日期範圍查,日期就不該是屬性而該是節點」。
- **抽取當下無法預知,業界靠「兩形式並存」**:literal 是節點的「尚未解析」形式
  (Wikidata `P50`/`P2093`、schema.org 容忍 Text)。

| 系統 | 值會變節點嗎 | 誰決定 · 何時 | 分數字/文字? | 矛盾處理 |
| --- | --- | --- | --- | --- |
| cognee | 一定會(結構上沒有屬性可放) | 寫死在 code | 否 | 無(寫入時) |
| LlamaIndex | 預設會(`1982` 變成節點) | 使用者 schema | 否 | 無 |
| Neo4j / LangChain | 屬性預設關閉 → 實際上會 | 使用者 config | 否 | 無(靜默丟棄) |
| GraphRAG | 不會,但烤進散文描述 | LLM,限型別白名單 | 否 | 無(claim 預設關閉) |
| Wikidata | 每個屬性事先宣告、永久不可改 | 人,建屬性時 | 否 | rank + references |
| RDF / Neo4j 指南 | 依「要不要談論 / 跨它查」 | 建模者,可重構 | 否 | 保留競爭陳述 |

## 3 · 已定案的決策

- [x] **矛盾偵測不做。** 系統不該替使用者判斷誰對誰錯;需求未明朗,做了吃力
      不討好。頁面只忠實列出「哪份文件說了什麼」,判斷留給讀者。
      (LLM 系四家也沒有一家在寫入時做。)
      ⚠️ 未來若要重開,先有明確需求;別把「這裡有事、自己去查」的告示牌加回來。
- [x] **分界不是型別,是「值 vs 東西」** — 而且**不預先判斷**。一律記
      「誰·什麼屬性·值(verbatim)」;值有沒有自己的節點,由「有沒有文件把它
      當主體談過」決定(= 既有 entity 規則,無 mention 即不存在)。
- [x] **qualifier 分兩層**(Wikidata):`unit` 屬於**值本身**;`period` 屬於
      **這句陳述**。
- [x] **有主語就用主語綁定,共現猜測整段刪除。** 兩套規則並存必有一套是假的。
- [x] **值一律 verbatim**,不解析成數字。顯示時單位只出現一次
      (值已含單位就不再補)。
- [x] **不做屬性白名單**(同 kind 的理由:預先寫死的清單只會往昂貴的方向錯)。

## 5 · Slice 二 — 抽取端統一(#630 / PR #631)

**問題**:`extract.py` 的 prompt 第一句寫死「Extract every metric that carries a
**numeric value**」,所以「X 的 recipe 是 PPOOIXUX」整條進不來 → 被關係抽取撿走
→ 值長出自己的節點、跑去跟別的名字做同名合併。第二個問題:claim **沒有主語**,
只能靠「同一張投影片」猜它屬於誰。

- [x] **P1** 抽取:任何值,而且每句都要說出主語。
      *一條測試盯著 prompt 本文,確保 `numeric` 不會回來。*
- [x] **P2** schema:`subject`/`norm_subject` + `norm_value`(都用
      `norm_surface` — 與實體名字同一套規則,這是「值日後能對上實體」的前提);
      `metric` → `attribute`;Schema v3。
- [x] **P3** 綁定:陳述落在**它指名的東西**上;共現猜測刪除。
      *測試挑「新舊規則給相反答案」的情境。*
- [x] **P4** 合流:一次 prompt 抽 mentions / aliases / relationships /
      attributes,每個 chunk 從 2 次 LLM 降為 1 次。
- [x] **P5/P6** 值升格:值被文件當主體談過 → 它就是 entity → 反查
      「誰把它當值用」(`value_of`)。API + 前端「誰用了它」+ 檔案卡。
- [x] **P7**:真模型驗收 + 單位重複的缺陷修正;模型不查圖的問題移交 slice 三解決

### 真模型驗收(qwen3-14b)


值裡沒有單位才補)。

## 6 · Slice 三 — 自動注入檔案卡(#633)

**問題**(#630 P7 實測):qwen3-14b 問到**它自己認得的詞**(「回焊爐是什麼?」)
就不查圖,直接用通用知識回答,而知識庫裡明明有它的 recipe 與良率。
四次嘗試都沒扳過來:兩種措辭、消除 `ask_wiki` 描述與檔案卡搶同一種問題的衝突、
工具清單改成便宜的排前面。問陌生代號(「PPOOIXUX 被哪些機台使用?」)則每次
都路由正確。

**根因**:模型用「我認不認得這個詞」決定要不要查,而不是「這是不是內部問題」。
措辭解不了 → **改成不問模型**:名字對得上就把檔案卡直接送進 context
(與 #106 glossary 區塊同一個位置、同一個時機)。

### 實測數據(全部量過)

| | 結果 |
| --- | --- |
| 記憶體:名字→ids(10 萬個名字,2% 同名,值一律 tuple) | **22 MB**(pod 上限 1 GiB 的 2%) |
| 每則訊息比對(set 交集 + 邊界驗證) | **13–20 µs**(1k → 500k 名字幾乎不變) |
| 命中後取完整資料(主鍵) | **0.3 ms** |
| ~~用名字查資料庫~~(否決) | **2849 ms** @ 4 萬筆(全掃) |
| ~~每則訊息撈全表比對~~(否決,glossary 現行做法) | **2976 ms** @ 4 萬筆 |

> 被否決的第二條正是 glossary 今天的做法,也是本 slice 原本打算沿用的實作 —
> 數字出來才改方向。

### 設計要點

- **不斷詞**。拿名字掃問句,不是從問句切名字 — 我們有詞表,所以是字串搜尋
  不是分詞;CJK 本來也沒有詞邊界可切。
- **交集在前、邊界規則在後**。set 只看字串相等,`m4` 會命中 `m40` 的子字串;
  倖存的那幾個再用 glossary 既有的邊界規則驗證(ASCII 有詞邊界保護,CJK 允許
  嵌入)。
- **一個名字對多個 id**(多 pod 併發建立、合併留下的 tombstone、跨 collection
  同名)。值**一律 tuple**,不用「有時字串有時 tuple」 — 兩種型別並存必有一處
  漏判;tuple 不可變,共用快取不會被讀取端手滑改掉。
- **只注入「有事實可講」的 entity**(有屬性陳述、或被別人當值用的)。這同時
  解掉「機台/良率這種通用短名每題都命中」與「注入了也沒用」。
- **同名時明說**「這個名字對應 N 個東西」,絕不默默挑第一個。
- 注入只加在**交給模型的那份訊息**上,持久化的訊息保持乾淨(同 #106)。
- **快取過期只影響自動注入**,不影響 agent 主動呼叫查圖工具 → 降級路徑始終
  存在,TTL 兜底就夠,不需要跨 pod 即時失效。

### Phases

- [x] **P1** 名字索引:`name → tuple[id]`、最短長度門檻、硬上限 + `dropped`
      計數(絕不靜默截斷)
- [x] **P2** 比對:問句 → 候選 → set 交集 → 邊界驗證
- [x] **P3** 注入區塊:精簡卡渲染 + 三道界(entity 數 / 每個的陳述數 / 字元數)
      + 同名揭露
- [x] **P4** 接線:KB chat turn(#106 的同一處);持久化訊息不變;
      索引快取(TTL + 重建失敗續用舊的)
- [x] **P5** 同名缺陷:`entity_card` 不再默默取第一個
- [x] **P6** 驗收(見下)

### 驗收結果(qwen3-14b,真模型)

| 情境 | 結果 |
| --- | --- |
| 「回焊爐是什麼?它用什麼 recipe?」(四次失敗的那句) | ✅ 答出 **POR recipe = PPOOIXUX**、位於產線三、與空洞有關,逐條附出處;並誠實區分「知識庫沒有直接定義」與一般理解 |
| 「PPOOIXUX 被哪些機台使用?」(反查) | ✅ 答出回焊爐(deck-A)+ 產線三(deck-B),且**零工具呼叫** — 注入就夠了 |
| 「什麼是傅立葉轉換?」(圖上沒有的東西) | ✅ 不受影響,注入區塊為空 |

延遲(5000 個 entity):

| | |
| --- | --- |
| 冷啟動建索引 | 262 ms(每 TTL 一次) |
| 命中的訊息 | 297 ms |
